# Спецификации прецедентов (детальные)

Ниже приведены детальные спецификации трёх ключевых прецедентов. Они соответствуют
фактически реализованным сценариям системы.

---

## UC-5. Запись на тест-драйв

| Поле | Значение |
|------|----------|
| **Актор** | Клиент |
| **Цель** | Записаться на пробную поездку на выбранном автомобиле |
| **Предусловия** | Клиент авторизован; автомобиль доступен (статус `IN_STOCK`) |
| **Постусловия (успех)** | Создана запись со статусом `PENDING`; менеджеры получили уведомление |
| **Триггер** | Нажатие «Записаться на тест-драйв» в карточке авто |

**Основной поток:**
1. Клиент открывает карточку автомобиля и выбирает «Записаться на тест-драйв».
2. Система открывает форму: дилерский центр, дата, время, контактный телефон, комментарий.
3. Клиент заполняет поля; система валидирует ввод **на лету** (дата не в прошлом, телефон по шаблону).
4. Клиент нажимает «Записаться».
5. Система проверяет доступность автомобиля и отсутствие пересечения слота (интервал ≥ 1 ч).
6. Система создаёт запись (`PENDING`), уведомляет клиента и менеджеров.
7. Система показывает подтверждение и возвращает клиента к списку заявок.

**Альтернативные потоки:**
- 3a. Поле некорректно → поле подсвечивается, под ним выводится конкретная ошибка, кнопка отправки не выполняет запрос.

**Исключения:**
- 5a. Автомобиль недоступен (`SOLD`/`RESERVED`/`UNAVAILABLE`) → ошибка 422 «Автомобиль недоступен для тест-драйва».
- 5b. Слот занят → ошибка 422 «Выбранный интервал занят».

**Бизнес-правила:** BR-3 (непересечение слотов), BR-5 (только доступные авто).

## UC-7. Оформление покупки

| Поле | Значение |
|------|----------|
| **Актор** | Клиент |
| **Цель** | Оформить заказ на покупку автомобиля (полная оплата или рассрочка) |
| **Предусловия** | Клиент авторизован; автомобиль `IN_STOCK`; нет активного заказа на это авто |
| **Постусловия (успех)** | Создан заказ `PENDING`; автомобиль переведён в `RESERVED`; выполнен переход к оплате |
| **Триггер** | Нажатие «Купить» / «Оформить рассрочку» в карточке авто |

**Основной поток:**
1. Клиент выбирает способ оплаты: полная или рассрочка.
2. *(рассрочка)* Клиент вводит первоначальный взнос и срок; система рассчитывает
   ежемесячный платёж и переплату (`«include»` UC-6).
3. Клиент подтверждает оформление.
4. Система проверяет доступность авто и отсутствие активного заказа.
5. Система создаёт заказ (`PENDING`), резервирует автомобиль (`RESERVED`), уведомляет
   клиента и менеджеров.
6. Система **сразу открывает экран оплаты** созданного заказа (`«include»` UC-8).

**Исключения:**
- 4a. Авто недоступно → 422 «Автомобиль недоступен для покупки».
- 4b. Уже есть активный заказ на это авто → 422 «По этому автомобилю уже оформлен активный заказ».
- 2a. Взнос ≥ стоимости автомобиля → 422 «Первоначальный взнос должен быть меньше стоимости».

**Бизнес-правила:** BR-1 (одна продажа), BR-2 (один активный заказ), BR-4 (взнос < цены).

## UC-8. Оплата заказа картой

| Поле | Значение |
|------|----------|
| **Актор** | Клиент |
| **Цель** | Оплатить заказ банковской картой (имитация эквайринга) |
| **Предусловия** | Заказ принадлежит клиенту и находится в статусе `PENDING`/`CONFIRMED` |
| **Постусловия (успех)** | Платёж `SUCCEEDED`; заказ переведён в `PAID`; уведомления отправлены |
| **Триггер** | «Оплатить» на экране заказа или сразу после оформления покупки |

**Основной поток:**
1. Система создаёт платёж (`PENDING`) для заказа; **сумма рассчитывается на сервере**
   (полная цена либо первоначальный взнос по рассрочке).
2. Клиент вводит реквизиты карты (номер, срок, CVC, держатель); форма валидирует номер по алгоритму Луна.
3. Клиент нажимает «Оплатить».
4. Сервер проверяет карту (Луна, срок), маскирует номер, имитирует ответ банка.
5. При успехе: платёж → `SUCCEEDED`, заказ → `PAID` (в одной транзакции); уведомления
   клиенту и менеджерам; показывается экран-чек.

**Альтернативные/исключительные потоки:**
- 4a. Тестовая карта-отказ (`4000 0000 0000 0002`) → платёж `FAILED`, заказ не оплачивается, показывается ошибка, доступна повторная попытка.
- 4b. Некорректный номер/истёкший срок → 422 с конкретным сообщением.
- 1a. Заказ уже оплачен → 422 «Этот заказ уже оплачен».
- Повторное подтверждение успешного платежа не приводит к повторному списанию (идемпотентность).

**Бизнес-правила:** BR-6 (сумма считается сервером), BR-7 (идемпотентность платежа),
BR-8 (полный номер карты не хранится — PCI DSS).
