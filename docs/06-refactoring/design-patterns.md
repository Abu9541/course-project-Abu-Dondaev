# Применённые паттерны проектирования (Этап 6)

## 1. Паттерны корпоративных приложений (Фаулер) — обязательные

### Data Mapper (обязательно) ✅

Отделяет доменные объекты от способа их хранения. Реализован **двухуровнево**:

1. **Доступ к данным** — Spring Data JPA репозитории (`foundation/*Repository`): Hibernate
   автоматически отображает сущности на таблицы; рукописный SQL для CRUD отсутствует.
2. **Преобразование Entity ↔ DTO** — слой `mapper/*Mapper` (`@Component`): например,
   `OrderMapper.toResponse(Order) : OrderResponse`. Это изолирует внутреннюю модель от
   контракта API и исключает «протечку» ленивых связей при сериализации.

```java
@Component
public class PaymentMapper {
    public PaymentResponse toResponse(Payment p) {
        return new PaymentResponse(p.getId(), p.getProviderPaymentId(),
                p.getOrder().getId(), p.getAmount(), p.getMethod(),
                p.getStatus(), p.getMaskedCard(), p.getCreatedAt());
    }
}
```

### Identity Map (обязательно) ✅

Гарантирует, что в пределах одной сессии каждому ряду БД соответствует **один** объект в
памяти. Обеспечивается **контекстом персистентности Hibernate** (first-level cache): в
рамках транзакции повторное обращение к сущности по id возвращает тот же экземпляр, без
повторного запроса и без рассинхронизации. Транзакционные границы заданы
`@Transactional` на сервисах.

### Lazy Load (рекомендуется) ✅

Связанные объекты загружаются по требованию: `@ManyToOne`/`@OneToMany` помечены `LAZY`
(кроме `Vehicle.brand` — `EAGER`, т.к. марка нужна почти всегда). Чтение с отдачей наружу
выполняется в `@Transactional(readOnly = true)`, что обеспечивает доступность ленивых
связей на момент маппинга в DTO.

### Unit of Work (через транзакции) ✅

Группа изменений фиксируется как единое целое транзакцией сервисного метода. Примеры:
- Оплата: `Payment → SUCCEEDED` **и** `Order → PAID` — атомарно.
- Продажа: `Order → COMPLETED`, `Vehicle → SOLD`, удаление авто из избранного — атомарно.

## 2. Паттерны GoF и архитектурные приёмы

| Паттерн | Где применён |
|---------|--------------|
| **Factory Method** | `InstallmentPlan.calculate(...)` — фабрика плана рассрочки |
| **Singleton** | Бины Spring (сервисы, репозитории) — один экземпляр на контейнер |
| **Dependency Injection** | Конструкторное внедрение на сервере; `AppContainer` (Service Locator) на клиенте |
| **Template Method** | Spring Data: типовые операции репозитория переопределяются производными запросами |
| **Strategy** | Выбор поведения по `PaymentType`/`OrderStatus` (полная оплата vs рассрочка) |
| **State** | Конечные автоматы статусов (`Order`, `TestDrive`, `Payment`) с проверкой переходов |
| **Observer (реактивность)** | `StateFlow` ↔ Compose: UI подписан на состояние ViewModel |
| **Adapter** | Retrofit-интерфейс `ApiService` адаптирует REST к вызовам методов клиента |
| **DTO** | Отдельные объекты передачи данных по сети |
| **Facade** | Репозитории клиента (`*Repository`) скрывают детали сети/кэша от ViewModel |

## 3. Принципы SOLID

- **S** — каждый слой и класс имеют единственную ответственность (контроллер ≠ логика ≠ доступ к данным).
- **O** — расширение через новые `enum`-значения и бизнес-методы без модификации существующих.
- **L** — реализации сервисов взаимозаменяемы за интерфейсами `*Service`.
- **I** — узкие интерфейсы сервисов/репозиториев по доменам.
- **D** — зависимость от абстракций (интерфейсов), а не от конкретных классов.
