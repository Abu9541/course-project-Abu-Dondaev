# Архитектурные решения (ADR)

Реестр ключевых архитектурных решений в формате ADR (Architecture Decision Record):
контекст → решение → статус → последствия.

---

### ADR-1. Адаптация PCMEF для клиент-серверного мобильного приложения

- **Контекст.** Траектория В требует мобильный клиент + сервер на Java. PCMEF исходно
  описывает один процесс.
- **Решение.** Слои C, M, E, F размещены на сервере (Spring Boot); слой P — на клиенте
  (Android). Граница P↔C проходит по REST/JSON.
- **Статус.** Принято.
- **Последствия.** (+) Чёткое разделение, независимое масштабирование, переиспользование
  API. (−) Сетевое взаимодействие требует обработки офлайна и ошибок на клиенте.

---

### ADR-2. Stateless-аутентификация на JWT

- **Контекст.** Мобильный клиент, REST API, требование RBAC.
- **Решение.** JWT (jjwt) + Spring Security; токен хранится на клиенте в DataStore и
  добавляется перехватчиком OkHttp; роли проверяются через `@PreAuthorize`.
- **Статус.** Принято.
- **Последствия.** (+) Не нужны серверные сессии, горизонтальное масштабирование.
  (−) Нужна аккуратная защита и хранение токена; отзыв токена сложнее (решается коротким сроком жизни).

---

### ADR-3. PostgreSQL + Spring Data JPA (Hibernate)

- **Контекст.** Реляционные данные (3НФ), связи, целостность; требование Data Mapper.
- **Решение.** PostgreSQL как СУБД; доступ — через репозитории Spring Data JPA.
- **Статус.** Принято.
- **Последствия.** (+) Data Mapper и Identity Map «из коробки», декларативные запросы,
  транзакции. (−) Зависимость от ORM-абстракций; для сложных запросов — JPQL/`@Query`.

---

### ADR-4. REST API с документацией OpenAPI

- **Контекст.** Контракт между клиентом и сервером должен быть стабильным и
  документированным.
- **Решение.** REST + springdoc-openapi (Swagger UI); стандартные HTTP-статусы; DTO для
  ввода/вывода (отделены от сущностей).
- **Статус.** Принято.
- **Последствия.** (+) Самодокументируемый API, удобство интеграции и проверки.
  (−) Дополнительный слой DTO и мапперов.

---

### ADR-5. Имитация платёжного шлюза вместо реального эквайринга

- **Контекст.** Реальный эквайринг невозможен в учебном проекте (мерчант, PCI, движение денег).
- **Решение.** Серверная имитация шлюза с реальным протоколом: создание платежа
  (`PENDING`) → подтверждение (`SUCCEEDED`/`FAILED`); сумма считается на сервере;
  идемпотентность; хранится только маска карты.
- **Статус.** Принято.
- **Последствия.** (+) Архитектура совместима с переходом на реальный шлюз
  (ЮKassa/CloudPayments) при минимальных правках; демонстрирует транзакции и обработку
  ошибок. (−) Не проводит реальные платежи.

### ADR-6. Офлайн-кэш каталога на Room

- **Контекст.** Нестабильная сеть у мобильного клиента; требование локального кэширования.
- **Решение.** Первая страница каталога кэшируется в Room; при отсутствии сети
  показываются сохранённые данные с офлайн-баннером.
- **Статус.** Принято.
- **Последствия.** (+) Базовый просмотр доступен офлайн. (−) Кэш ограничен каталогом
  (не все экраны); возможна неактуальность данных до синхронизации.

---

### ADR-7. Ручной DI (Service Locator) на клиенте

- **Контекст.** Нужна простая, прозрачная сборка зависимостей без тяжёлого фреймворка.
- **Решение.** `di.AppContainer` создаёт репозитории и менеджеры в `Application`;
  ViewModel получают зависимости через фабрику `vmFactory`.
- **Статус.** Принято.
- **Последствия.** (+) Прозрачность, нет генерации кода и долгой сборки (как с Hilt/Dagger).
  (−) Ручная проводка зависимостей; для крупного проекта предпочтительнее DI-фреймворк.

---

### ADR-8. UI на Jetpack Compose (Material 3)

- **Контекст.** Современный декларативный UI, поддержка состояний и тем.
- **Решение.** Jetpack Compose + Material 3; состояние — ViewModel + StateFlow; навигация —
  Navigation Compose с роле-зависимым нижним меню.
- **Статус.** Принято.
- **Последствия.** (+) Меньше шаблонного кода, реактивный UI, лёгкое переключение темы.
  (−) Требует актуальных версий и совместимости Compose-компилятора с Kotlin.
