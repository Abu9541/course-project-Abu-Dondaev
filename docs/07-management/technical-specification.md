# Техническое задание (ТЗ)

## 1. Общие сведения

- **Наименование системы:** Информационная система «Автосалон» (мобильное приложение).
- **Назначение:** цифровой канал подбора, тест-драйва и покупки автомобилей с серверной
  частью и ролевым доступом.
- **Заказчик/автор:** Дондаев А. У.-П., ПИЖ-б-о-23-1, СКФУ.
- **Основание:** курсовой проект по дисциплине «Программная инженерия» (траектория В).

## 2. Назначение и цели создания

См. [паспорт проекта](../00-initiation/project-passport.md). Кратко: перенести подбор,
тест-драйв и покупку авто в мобильное приложение; упорядочить обработку заявок персоналом;
дать руководству аналитику.

## 3. Требования к системе

### 3.1. Требования к функциональным характеристикам

Перечень подсистем и функций (детально — в [трассировке](../01-requirements/traceability-matrix.md)):

| Подсистема | Функции |
|------------|---------|
| Аутентификация | Регистрация, вход (JWT), роли CLIENT/MANAGER/ADMIN |
| Каталог | Просмотр, поиск, фильтрация (марка/кузов/цена), пагинация, офлайн-кэш |
| Карточка авто | Характеристики, фотогалерея, отзывы, средний рейтинг |
| Тест-драйв | Запись, проверка слота, обработка персоналом |
| Покупка | Полная оплата / рассрочка (калькулятор), резервирование |
| Оплата | Имитация эквайринга: создание → подтверждение, идемпотентность |
| Заявки/заказы | Статусная модель, фильтр по статусу, отмена |
| Уведомления | События клиенту и персоналу, счётчик непрочитанных |
| Избранное, отзывы | Управление избранным, отзывы (1–5) |
| Аналитика | Панель: наличие, продажи, выручка, заявки |
| Администрирование | CRUD каталога, управление пользователями, проданные авто |
| Профиль/настройки | Редактирование, тема, поддержка, соглашение |

### 3.2. Требования к архитектуре (PCMEF)

- Слоистая архитектура **P–C–M–E–F**; зависимости строго сверху вниз, без циклов.
- Слои общаются через интерфейсы (`*Service`, `*Repository`).
- Подробно — в [docs/02-architecture](../02-architecture/pcmef-architecture.md).

### 3.3. Требования к данным

- СУБД PostgreSQL, нормализация до 3НФ, целостность (PK/FK/UNIQUE/CHECK), индексы.
- Подробно — в [docs/03-database](../03-database/database-design.md).

### 3.4. Требования к безопасности

- Аутентификация по JWT; ролевое разграничение (RBAC) на уровне эндпоинтов.
- Хеширование паролей BCrypt; хранение токена на клиенте в DataStore.
- Защита ПДн (152-ФЗ): согласие на обработку, хранение только маски карты (PCI DSS).

### 3.5. Требования к надёжности

- Время восстановления после отказа: перезапуск контейнеров `docker compose up` (минуты);
  данные сохраняются в томе PostgreSQL.
- Сохранность при сбоях: транзакции `@Transactional` гарантируют атомарность операций
  (оплата, продажа); согласованность статусов.
- Обработка ошибок сети на клиенте: состояния загрузки/ошибки/офлайна, безопасные вызовы.

### 3.6. Требования к интерфейсу

- Нативный Android (Compose, Material 3); не менее 5 экранов (реализовано 18).
- Поддержка светлой/тёмной/системной темы.
- Обработка состояний (загрузка/ошибка/пусто), моментальная валидация форм.

### 3.7. Требования к развёртыванию

- Сервер + БД: Docker + docker-compose (backend на порту 8081, PostgreSQL).
- Клиент: сборка APK, установка на устройство/эмулятор (minSdk 26).
- Подробно — в [руководстве администратора](admin-guide.md).
