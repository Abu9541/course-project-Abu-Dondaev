## 2. Бизнес-контекст

**Проблема.** Покупка автомобиля традиционно требует личного посещения салона: клиент не
может заранее изучить актуальный ассортимент, сравнить комплектации и цены, рассчитать
рассрочку и записаться на тест-драйв в удобное время. Менеджеры обрабатывают заявки
вручную (звонки, бумажные журналы), что приводит к потерям обращений и низкой
прозрачности воронки продаж.

**Решение.** Клиент-серверная информационная система с нативным мобильным приложением:
единый каталог автомобилей с фильтрацией и поиском, онлайн-оформление покупки (полная
оплата или рассрочка с калькулятором), имитация оплаты картой, запись на тест-драйв,
избранное, отзывы, push-подобные уведомления. Для персонала — обработка заявок и заказов,
аналитическая панель; для администратора — управление каталогом и пользователями.

## 3. Цели проекта

1. **Повысить доступность услуг салона** — перенести подбор авто, запись на тест-драйв и
   оформление покупки в мобильное приложение (24/7, без визита в салон).
2. **Ускорить и упорядочить обработку обращений** — единый поток заявок и заказов со
   статусной моделью и уведомлениями для менеджеров.
3. **Обеспечить прозрачность сделки для клиента** — онлайн-расчёт рассрочки, понятные
   статусы заказа, история заявок.
4. **Дать руководству инструмент контроля** — аналитическая панель (наличие, продажи,
   выручка, заявки).

## 4. Ключевые показатели успеха (KPI)

| Показатель | Целевое значение |
|------------|------------------|
| Доля заявок/заказов, оформленных через приложение | ≥ 60 % от общего числа |
| Среднее время реакции менеджера на новую заявку | ≤ 2 часов (за счёт уведомлений) |
| Доля «потерянных» обращений | ≤ 5 % |
| Доступность каталога без сети (офлайн-кэш) | базовый просмотр доступен 100 % времени |
| Покрытие серверного ядра модульными тестами | > 40 % |

## 5. Границы проекта (Scope)

**Входит:** каталог и карточки авто, поиск/фильтрация, покупка (полная/рассрочка),
имитация онлайн-оплаты картой, тест-драйв, избранное, отзывы, уведомления, аналитика,
администрирование каталога и пользователей, ролевая модель, офлайн-кэш каталога.

**Не входит (за рамками курсового):** реальная интеграция с банковским эквайрингом и
движение реальных денег, интеграция с 1С/CRM салона, доставка/логистика, iOS-клиент,
электронный документооборот договоров.

## 6. Ключевые риски

| Риск | Стратегия реагирования |
|------|------------------------|
| Сжатые сроки (один семестр) | Поэтапная разработка по МУ, приоритет обязательного ядра PCMEF |
| Недоступность реального эквайринга | Имитация платёжного шлюза с сохранением архитектуры (intent → confirm) |
| Нестабильная сеть у клиента | Локальное кэширование каталога (Room), обработка состояний загрузки/ошибки/офлайна |
| Несанкционированный доступ к данным | JWT-аутентификация, ролевое разграничение (RBAC), BCrypt-хеширование паролей |
| Рост сложности кода | Строгая слоистая архитектура PCMEF, модульные тесты, статический анализ |

## 7. Технологический стек (кратко)

- **Клиент:** Kotlin, Jetpack Compose (Material 3), Retrofit/OkHttp, Room, DataStore, Coil, Navigation Compose.
- **Сервер:** Java 17, Spring Boot (Web, Data JPA, Security), JWT (jjwt), Bean Validation, springdoc-openapi.
- **БД:** PostgreSQL (3НФ).
- **Развёртывание:** Docker + docker-compose.
- **Качество:** JUnit 5 + Mockito + JaCoCo.
