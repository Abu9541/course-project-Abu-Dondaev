# SWOT-анализ

Анализ внутренних и внешних факторов проекта/продукта ИС «Автосалон».

| | **Помогают достижению цели** | **Мешают достижению цели** |
|---|------------------------------|-----------------------------|
| **Внутренние факторы** | **S — Сильные стороны** | **W — Слабые стороны** |
| | • Строгая слоистая архитектура PCMEF, тестируемость<br/>• Нативный Android-клиент (быстрый UX, доступ к API устройства)<br/>• Готовый REST API с OpenAPI и JWT-безопасностью<br/>• Офлайн-кэш каталога<br/>• Расширяемая доменная модель | • Один разработчик, ограниченные сроки (1 семестр)<br/>• Имитация платежей вместо реального эквайринга<br/>• Только Android (нет iOS)<br/>• Нет интеграции с учётными системами салона (1С/CRM) |
| **Внешние факторы** | **O — Возможности** | **T — Угрозы** |
| | • Рост спроса на онлайн-покупку авто<br/>• Масштабирование на сеть салонов (мультидилерство)<br/>• Подключение реального эквайринга и доставки<br/>• Публикация в Google Play, push-уведомления (FCM) | • Высокая конкуренция агрегаторов (Авто.ру, Дром)<br/>• Требования к защите персональных данных (152-ФЗ)<br/>• Изменения правил банковского эквайринга<br/>• Зависимость от стабильности сети у клиента |

## Стратегические выводы (перекрёстный анализ)

- **SO (сила + возможность):** на базе чистой архитектуры и готового API быстро
  масштабировать систему на несколько салонов и подключить реальную оплату.
- **WO (слабость + возможность):** имитацию платежей спроектировать так, чтобы переход
  на реальный эквайринг (ЮKassa/CloudPayments) требовал минимальных изменений — что и
  реализовано (модель «создание платежа → подтверждение»).
- **ST (сила + угроза):** JWT + RBAC + BCrypt и хранение токена в защищённом
  DataStore снижают риски ИБ и помогают соответствовать 152-ФЗ.
- **WT (слабость + угроза):** ограниченные сроки и конкуренция требуют фокуса на
  обязательном ядре и ключевых сценариях (подбор → тест-драйв → покупка → оплата).
